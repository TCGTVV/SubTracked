# Rust Build Pipeline: Typgenerierung + Compile-time-SQL

## Rust→TS-Typgenerierung (seit 2026-07-14)

`ts-rs 12` (`serde-compat`-Feature), optionale Dependency hinter Cargo-Feature `ts-rs-export` (vorher unbedingte Dependency, kompilierte in jeden Release-Build mit). `Subscription`/`Account`/`Income`/`PriceHistoryEntry` in `src-tauri/src/db.rs` tragen `#[cfg_attr(feature = "ts-rs-export", derive(TS))]` + `ts(export, export_to = "../../src/generated/<Name>.ts")`; i64-Felder brauchen explizit `ts(type = "number")`/`"number | null"` (ts-rs mapt i64 sonst auf `bigint`, Tauri-IPC transportiert aber JSON-Zahlen — ohne Override bricht Zahlen-Arithmetik im Frontend unbemerkt). Vitest-Test `src/lib/generated-bindings.test.ts` prüft jede generierte Datei auf Substring "bigint". Export läuft über `db::export_bindings_*`-Hidden-Tests, ausgelöst durch `cargo nextest run --features ts-rs-export export` in `src-tauri/` (default `cargo nextest run` hat dadurch 118 statt 122 Tests). `src/types.ts` importiert generierte Typen und verschärft `interval`/`cancelMode`/`cancelPeriodUnit` (in Rust nur validierte Strings) per `Omit`+Intersection auf handgepflegte literale Unions `Interval`/`CancelMode`/`CancelUnit`. `src/generated/**` + `src-tauri/.sqlx/**` sind aus `biome.json`s `files.includes` ausgeschlossen (generiert, eigenes Format). CI prüft per `git diff --exit-code -- src/generated`, dass committete Bindings aktuell sind (kein Regenerieren in CI, reiner Diff-Check).

## Compile-time-verifizierte SQL (seit 2026-07-14)

Alle statischen Queries über `sqlx::query!`/`query_as!` (85 Stellen über `commands.rs`/`reminders.rs`/`backup.rs`/`db_backup.rs`/`csv_export.rs`/`csv_import.rs`/`csv_reconcile.rs`/`validation.rs`, inkl. Test-Module) — Tippfehler in Spalten-/Tabellennamen brechen den Build statt erst zur Laufzeit. Genuines dynamisches SQL bleibt bewusst bei `sqlx::query()`: Backup-Restore-DELETEs (5 explizite `query!`-Aufrufe statt Schleife über `&str`-Array), `VACUUM INTO` mit interpoliertem Pfad in `db_backup.rs`, zwei Tests mit synthetischer Tabelle `t` (existiert nicht im echten Schema). SQLite-Pragmas brauchen klassische `PRAGMA x`-Syntax, nicht `pragma_x()`-Tabellenfunktionen (scheitern an sqlx' Nullability-Introspektion: "no such table column"). Normale Builds brauchen keine DB: `src-tauri/.cargo/config.toml` setzt `SQLX_OFFLINE=true` (force=false), committetes `src-tauri/.sqlx/` (84 Cache-Dateien) liefert die Query-Metadaten. Für neue/geänderte Queries lokal: `sqlx-cli` (`cargo install sqlx-cli --no-default-features --features sqlite,rustls`), `src-tauri/.env` mit `DATABASE_URL=sqlite://.dev.db` (gitignored), einmalig `cargo sqlx database create && cargo sqlx migrate run --source ./migrations`, danach nach jeder Query-Änderung `cargo sqlx prepare -- --tests` (das `--tests` ist nötig, sonst fehlen die `#[cfg(test)]`-Queries im Cache) und den `.sqlx/`-Diff mitcommitten. CI braucht kein `DATABASE_URL` — der `.cargo/config.toml`-Default greift automatisch.

## Gotcha: sqlx::query!-Literalzwang

`sqlx::query!`/`query_as!` verlangen zwingend ein String-**Literal** an der Aufrufstelle — weder ein `const &str` noch ein `macro_rules!`-Wrapper (nicht eager expandiert) funktionieren als Umweg. Deshalb ist die 19-Spalten-Subscription-Query nicht auf einen Aufrufer reduzierbar (nur auf `fetch_subscriptions` in `db.rs`), und die Backup-Restore-DELETE-Schleife musste zu 5 Einzelaufrufen entrollt werden.
