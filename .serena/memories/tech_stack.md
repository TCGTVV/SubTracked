# Tech-Stack

- **Tauri v2** (Rust-Kern, System-WebView). Targets: Windows, Linux, macOS.
- **Frontend:** React 19 + TypeScript 5.8, Vite 7, `@vitejs/plugin-react`.
- **UI-Stack (seit 2026-06-16, Overhaul abgeschlossen):** **Tailwind CSS v4** (`@tailwindcss/vite`) + **shadcn/ui** (new-york, baseColor neutral; Komponenten in `src/components/ui/`) auf **`radix-ui`-Umbrella** (`^1.5`, nicht die `@radix-ui/react-*`-Einzelpakete) + **lucide-react** (Icons). `clsx`+`tailwind-merge` als `cn()` in `src/lib/utils.ts`, `class-variance-authority` für Varianten, `tw-animate-css` für Enter/Exit-Animationen. Alias `@`→`src` in vite/vitest/tsconfig. Token-System + Fluid-`clamp()`-Tokens + Dark-Mode (`.dark`) in `src/index.css` (oklch). **Kein MUI/Emotion/Material-Symbols, kein `App.css` mehr** — alle Komponenten Token-basiert. `react-day-picker` ist **v10** (Calendar handgeschrieben gegen v10-API). Gotcha: `Select` steht auf `position="popper"` (item-aligned flackert beim Schließen in WebKitGTK); globale `[data-state="closed"]{animation-fill-mode:forwards}`-Regel in index.css fixt das Exit-Flicker aller Radix-Overlays.
- **DB:** SQLite via **eigenem `sqlx`-Pool** (`sqlx 0.9.0`, Features `runtime-tokio`, `sqlite`, `macros`, `migrate`). Pool wird im Tauri-`setup`-Block geöffnet und in `app.manage(AppState { db: pool })` abgelegt. Migrations laufen via `sqlx::migrate!("./migrations").run(&pool)` direkt nach dem Pool-Open. Frontend ruft `invoke<T>(...)` aus `@tauri-apps/api/core`; die DB-Schicht in Rust liegt in `src-tauri/src/{db,commands}.rs`. Seit 2026-06-06 ist das die **einzige** DB-Schicht — `tauri-plugin-sql` (Crate + `@tauri-apps/plugin-sql`) komplett raus. Compile-time-Query-Verifikation: siehe eigener Abschnitt weiter unten.
- **Date-Math:** `chrono 0.4` (`no-default-features`, Features `clock`, `std`) für den Reminder-Scheduler. Anker-additive Recurrence-Logik in `src-tauri/src/recurrence.rs` (parallel zur TS-Version in `src/lib/recurrence.ts`, die weiter für die Kontodeckungs-Ansicht im Frontend gebraucht wird).
- **Async-Tasks:** `tokio 1` (Feature `time`) für `tokio::time::sleep` im Reminder-Loop. tauri::async_runtime ist der Spawn-Wrapper.
- **Logging (seit 2026-06-07):** `tracing 0.1` + `tracing-subscriber 0.3` (Feature `env-filter`) + `tracing-appender 0.2`. Im Setup-Block werden zwei Layer initialisiert: stdout (sichtbar bei `pnpm tauri dev`) und ein Daily-Rolling-File-Appender im `app_log_dir()` (`~/.local/share/com.tcgtvv.subtracked/logs/` auf Linux, `~/Library/Logs/com.tcgtvv.subtracked/` auf macOS, `%LOCALAPPDATA%\com.tcgtvv.subtracked\logs\` auf Windows), max. 7 Tagesdateien. WorkerGuard wird via `app.manage(guard)` am Leben gehalten. Default-Level `info` via `EnvFilter`, override via `RUST_LOG=debug`. Macros: `tracing::info!`, `tracing::error!(error = %e, "…")` etc.
- **Notifications:** `tauri-plugin-notification ~2.3`. Aus Rust via `NotificationExt::notification().builder()...`. Permission-Check über `permission_state() == PermissionState::Granted`.
- **Autostart:** `tauri-plugin-autostart ~2.5` (nur Desktop, nicht Mobile — bereits per `cfg`-Gate in `Cargo.toml`).
- **Util-Frontend:** `date-fns ^4` (für `recurrence.ts` / `coverage.ts` / `format.ts`).
- **Paketmanager:** `pnpm@11.3.0` (deklariert via `packageManager`-Feld in `package.json`). Benötigt Node.js **≥ 22.13** wegen `node:sqlite`-Builtin. `pnpm-workspace.yaml` setzt `allowBuilds.esbuild: true` und `allowBuilds.lefthook: true` (lefthook braucht das postinstall-Script, das die Go-Binary downloadet — pnpm v11 blockt sonst aus Sicherheitsgründen).

## SQLite-Pfad

`subtracker.db` — im Tauri-`setup`-Block via `app.path().app_config_dir().join("subtracker.db")` geöffnet. Auf Linux: `~/.config/com.tcgtvv.subtracked/subtracker.db`. macOS: `~/Library/Application Support/com.tcgtvv.subtracked/`. Windows: `%APPDATA%\com.tcgtvv.subtracked\`. WAL-Mode aktiv (`SqliteConnectOptions::journal_mode(SqliteJournalMode::Wal)`); `create_if_missing(true)` legt die Datei bei frischer Installation an. Foreign-Key-Enforcement ist unter `sqlx-sqlite` per Default an — das `delete_subscription`-Command wickelt das daher in eine Transaktion (Reminders zuerst, dann Abo).

## Rust-Architektur (seit 2026-06-06)

- `src-tauri/src/lib.rs` — Tauri-Builder, Plugin-Registrierung, `invoke_handler!`, **tracing-Subscriber-Init** (stdout + rolling-Datei) ganz am Anfang des Setup-Blocks, Pool-Setup, Tray + Window-Verhalten, **Reminder-Scheduler-Spawn** (`tauri::async_runtime::spawn` mit initial-Check und `tokio::time::sleep(REMINDER_INTERVAL)`-Loop, `REMINDER_INTERVAL = 1h`).
- `src-tauri/src/db.rs` — `AppState { db: SqlitePool }` als Tauri-managed State, plus Rust-Structs `Subscription` / `Account` / `NewSubscription` mit `#[derive(sqlx::FromRow, Serialize, Deserialize)]` und `#[serde(rename_all = "camelCase")]` für die JSON-Bridge.
- `src-tauri/src/commands.rs` — neun `#[tauri::command(rename_all = "camelCase")]`-Funktionen: `list_subscriptions`, `list_accounts`, `add_subscription`, `delete_subscription`, `add_account`, `delete_account`, `count_subs_for_account`, `update_subscription`, `insert_reminder_if_new`.
- `src-tauri/src/recurrence.rs` — `next_due_date(anchor, interval, from)` mit anker-additiver Logik via `NaiveDate::checked_add_months(Months::new(k * step))`. Tests in `mod tests` (inkl. 31.-Anker-Drift, Quartal/Jahr, Schaltjahr-Klemmung, Fehlerpfad).
- `src-tauri/src/reminders.rs` — `run_reminder_check(pool, app_handle) -> Result<u32, String>`: liest aktive Subs, prüft pro Sub `notify`/Vorlauf-Fenster, `INSERT OR IGNORE INTO reminders` für Idempotenz, sendet Notification via `tauri-plugin-notification` wenn Permission granted.

## Rust→TS-Typgenerierung (seit 2026-07-14)

`ts-rs 12` (`serde-compat`-Feature, respektiert `#[serde(rename_all = "camelCase")]`/`#[serde(default)]`). `Subscription`/`Account`/`Income`/`PriceHistoryEntry` in `src-tauri/src/db.rs` tragen `#[derive(TS)]` + `#[ts(export, export_to = "../../src/generated/<Name>.ts")]`; i64-Felder haben explizit `#[ts(type = "number")]`/`"number | null"` (ts-rs mapt i64 sonst auf `bigint`, aber die Tauri-IPC transportiert JSON-Zahlen, keine BigInts — ohne den Override bricht jede Zahlen-Arithmetik im Frontend). Export läuft über einen von ts-rs generierten Hidden-Test pro Struct (`db::export_bindings_*`), ausgelöst durch `cargo test export` in `src-tauri/` — kein separates Build-Tool. `src/types.ts` importiert die generierten Typen aus `src/generated/` und verschärft `interval`/`cancelMode`/`cancelPeriodUnit` (in Rust nur validierte Strings, kein Enum) per `Omit`+Intersection auf die handgepflegten literalen Unions `Interval`/`CancelMode`/`CancelUnit`. `src/generated/**` und `src-tauri/.sqlx/**` sind bewusst aus `biome.json`s `files.includes` ausgeschlossen (generierte Dateien, eigenes Format).

## Compile-time-verifizierte SQL (seit 2026-07-14)

Alle statischen Queries über `sqlx::query!`/`query_as!` (85 Stellen über `commands.rs`, `reminders.rs`, `backup.rs`, `db_backup.rs`, `csv_export.rs`, `csv_import.rs`, `csv_reconcile.rs`, `validation.rs` inkl. Test-Module) — Tippfehler in Spalten-/Tabellennamen brechen jetzt den Build statt erst zur Laufzeit. Genuines dynamisches SQL bleibt bewusst bei `sqlx::query()`: Backup-Restore (`DELETE`-Liste war eine Schleife über ein `&str`-Array, jetzt 5 explizite `query!`-Aufrufe; INSERT-Schleifen über `Vec<T>` sind unproblematisch, da der Literal-String pro Aufruf gleich bleibt), `VACUUM INTO` mit interpoliertem Pfad in `db_backup.rs`, sowie zwei Tests, die eine synthetische Tabelle `t` anlegen (existiert nicht im echten Schema, kann nicht compile-time geprüft werden). SQLite-Pragmas brauchen die klassische `PRAGMA x`-Syntax, nicht `pragma_x()`-Tabellenfunktionen (letztere scheitern an sqlx' Nullability-Introspektion: "no such table column"). Normale Builds brauchen **keine** DB: `src-tauri/.cargo/config.toml` setzt `SQLX_OFFLINE=true` (force=false), das committete `src-tauri/.sqlx/`-Verzeichnis (84 Cache-Dateien) liefert die Query-Metadaten — verifiziert per Test ohne `DATABASE_URL`/Dev-DB. Für neue/geänderte Queries lokal: `sqlx-cli` (`cargo install sqlx-cli --no-default-features --features sqlite,rustls`), `src-tauri/.env` mit `DATABASE_URL=sqlite://.dev.db` (gitignored) + `cargo sqlx database create && cargo sqlx migrate run --source ./migrations` einmalig, danach `cargo sqlx prepare -- --tests` nach jeder Query-Änderung (das `--tests` ist nötig, sonst fehlen die `#[cfg(test)]`-Queries im Cache) und den `.sqlx/`-Diff mitcommitten. CI (`checks.yml`) braucht keine Anpassung — kein `DATABASE_URL` gesetzt, der `.cargo/config.toml`-Default greift automatisch.
